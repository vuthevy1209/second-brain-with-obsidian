## Recap

- Subagents can review code with fresh context before pushing
- `/commit-push-pr` handles commit, push, and PR creation in one step
- `claude --from-pr <N>` resumes a session linked to a PR
# Code Review

Claude Code provides built-in features to streamline code review and PR workflow: subagent reviews, `/commit-push-pr` skill, and `--from-pr` session linking.

## Review with a Subagent

Use a **subagent** for unbiased code review before pushing a PR:

- Runs in its own context window with fresh eyes
- Doesn't carry bias from the main agent that wrote the code
- Restrict to read-only tools (reviewer flags issues, doesn't edit)

![[6.Code Review-1777891582093.webp]]

Configure the reviewer subagent in your repo so the whole team uses the same standards.

### Example: Requesting a Subagent Review

```
You: "Use a subagent to review my changes before I push this PR"

Claude: [Launches code-reviewer subagent with read-only tools]

Subagent: "Reviewing changes in feature/user-auth branch...

Found 3 issues:
1. Line 45 in auth.ts: Missing input validation on user input
2. Line 89 in auth.ts: API key exposed in client-side code
3. Line 120: No error handling for failed login attempts

Recommendations: [provides specific fixes for each]"
```

## The /commit-push-pr Skill

Handles the full commit-to-PR flow in one step:

1. Creates the commit
2. Pushes to remote
3. Creates the PR

If you have a Slack MCP server configured with channels listed in CLAUDE.md, it automatically posts the PR link to your team's channel.

### Example: Using /commit-push-pr

```
You: "/commit-push-pr"

Claude: "Creating commit...
- Staging changed files: auth.ts, login.ts
- Commit message: 'feat: add user authentication with input validation'
- Pushing to origin/feature/user-auth...
- Creating PR...

PR created: https://github.com/your-repo/pull/123
Posted to #dev-team Slack channel."
```

## Session Linking with --from-pr

When Claude creates a PR via `gh pr create`, the session links to that PR automatically.

To resume work on a PR later (address review comments, fix failing build):

```bash
claude --from-pr <PR_NUMBER>
```

This picks up right where you left off.

### Example: Resuming a PR Session

```bash
# After reviewer requests changes on PR #123
$ claude --from-pr 123

Claude: "Resuming session for PR #123: 'feat: add user authentication'

Context restored:
- Original task: Implement user auth
- Reviewer comments: 3 issues found
- Build status: Failed (1 test)

What would you like to address first?"
```

## Workflow Diagram

```
┌─────────────────────────────────────────────────────────────┐
│                    CODE REVIEW WORKFLOW                     │
└─────────────────────────────────────────────────────────────┘

    [Write Code] 
         │
         ▼
    [Run Tests] ────→ (Fail?) ──→ [Fix & Retest]
         │                              │
        (Pass)                          │
         │                              │
         ▼                              │
    [Subagent Review] ◄─────────────────┘
         │
         │ (Issues found?)
         ├──→ (Yes) ──→ [Fix Issues] ──→ [Re-review]
         │
         │ (No issues)
         ▼
    [/commit-push-pr]
         │
         ├──→ Commit created
         ├──→ Push to remote
         └──→ PR created (+ Slack notification)
              │
              ▼
         [PR Review]
              │
              ├──→ (Changes requested) ──→ [claude --from-pr N] 
                               ──→ [Fix] ──→ [Re-push]
              │
              └──→ (Approved) ──→ [Merge]
```
