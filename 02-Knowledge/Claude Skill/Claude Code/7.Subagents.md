## Recap

- Keeping your context window clean is one of the best ways to stay productive with Claude Code.
- With subagents, you can run an agent in the background to handle the heavy lifting and return just the answer to your main context window.
- Read more: https://anthropic.skilljar.com/introduction-to-subagents

## How Subagents Work

![[7.Subagents-1777912308385.webp|459x376]]

Managing context in Claude Code is important.<mark style="background:rgba(3, 135, 102, 0.2)"> A lot of the context window gets consumed by things like tool calls exploring your codebase or running web searches for research.</mark> But, what Claude discovers during that exploration isn't always relevant to the main feature you're developing.

This is where subagents come in. Claude spawns a subagent to handle a task like "explore this codebase for me".

- The subagents runs in parallel with its own context window - does all the exploration work.
- And one finished, summarizes its findings and returns that summary back to Claude.

> [!check] Results
> Results: Heavy work happens in subagent, only the answer comes back → main context stays clean.

```
				┌─────────────────────────────────────────────────┐
				│           MAIN CONTEXT WINDOW                   │
				│  (Claude Code - your main conversation)         │
				│                                                 │
				│   User: "Search the codebase and fix X"         │
				│                                                 │
				│   ┌─ Decides to delegate ─────────────┐         │
				│   │                                   │         │
				│   ▼                                   ▼         │
				│  ┌─────────────────────────────────────┐        │
				│  │           SUBAGENT                  │        │
				│  │  (Isolated fresh context window)    │        │
				│  │                                     │        │
				│  │  • Reads many files                 │        │
				│  │  • Runs heavy searches              │        │
				│  │  • Uses tools independently         │        │
				│  │  • Does NOT bloat main context      │        │
				│  │                                     │        │
				│  │  Returns: short summary/result only │        │
				│  └─────────────────────────────────────┘        │
				│                                                 │
				│   ◄── Receives concise result  ──────────────── │
				│                                                 │
				│   Main continues with clean context             │
				│   "Here's the fix based on findings..."         │
				└─────────────────────────────────────────────────┘
				
				Key benefit: Heavy work happens in subagent,
				only the answer comes back → main context stays clean.
```

## Types of Subagents

- **Built-in subagents** — e.g., the default "research" subagent for common patterns
- **Custom subagents** — user-defined with custom system prompt, tool set, and permissions
- **Team subagents** — part of an agent team for parallel multi-agent workflows
	![[7.Subagents-1777917722678.webp]]

## When to Use

- Searching large codebases without filling context
- Running heavy file reads or analysis in the background
- Parallelizing independent research tasks
- Keeping the main conversation focused on the final answer

# Creating Your Own Subagent

Subagents are defined in Markdown files with YAML frontmatter. Run `/agents` then select "Create new agent".

![[7.Subagents-1777913686766.webp]]

- **Project:** create inside project => share team.
- **Personal:** create at root folder in personal machine.
#### Folder Structure

After you have created a custom subagent, Claude Code sets up two folders inside `.claude/`:

```
.claude/
├── agents/                # Where agent definitions live
│   └── your-agent-name.md  # The agent's prompt and config
└── agent-memory/           # Persistent memory storage per agent
    └── your-agent-name/    # Folder named exactly like the agent file
        └── MEMORY.md       # Index file where the agent logs learned facts over time
```

![[7.Subagents-1777917492478.webp|473x279]]
##### Agent Definitions `agents/`

This folder stores one `.md` file per custom agent. Each file contains:
- **YAML frontmatter** — agent metadata (name, description, model, color, memory scope)
- **Body** — the system prompt that defines wh<mark style="background:rgba(3, 135, 102, 0.2)">a</mark>t the agent does, its workflow, and guidelines

##### Agent Memory `agent-memory/`

- This folder mirrors the agents folder with one subfolder per agent (matching the agent filename without `.md`). 
- Inside each subfolder, the agent can write persistent memory files that survive across conversations. The `MEMORY.md` file acts as an index for all stored memories.

### Agent MD File Structure

Each agent file (e.g., `obsidian-note-proofreader.md`) follows this structure:

```markdown
---
name: "agent-name"                  # Unique identifier, used in Agent tool calls
description: "When to use this agent" # Shown in /agents UI; tells Claude when to spawn it
model: haiku                         # Optional: haiku | sonnet | opus (default: inherits from parent)
color: cyan                          # Optional: UI color for this agent
memory: project                      # Optional: project (shared via git) | user (local only)
---

System prompt goes here. This is the instruction given to the agent
when it is spawned. Define:
- The agent's role and responsibilities
- Step-by-step workflow
- Guidelines and constraints
- Output format
- How to use agent-memory (if applicable)
```
